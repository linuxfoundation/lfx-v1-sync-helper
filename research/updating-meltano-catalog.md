# Updating the Meltano catalog ConfigMap

Meltano uses a pre-generated catalog file to know each table's schema (field names, types, primary keys). Think of it as a **cache of all columns across the tables listed in `meltano.yml`**. Without it, Meltano would run its own schema discovery on every pipeline run, but it has no built-in way to persist that cache externally. The catalog ConfigMap solves this by storing the discovered schema in Kubernetes so it can be reused across runs.

The catalog is generated by connecting to real DynamoDB tables, so it is environment-specific and is **never committed to the repo** — it must be applied directly to each cluster.

## When to update

- A new table is added to `meltano.yml`
- Columns change in a source DynamoDB table

## Check the current catalog ConfigMap

Make sure your `kubectl` context is pointed at the target cluster before running any commands:

```bash
# Confirm which cluster you're targeting
kubectl config current-context

# View the current catalog ConfigMap
kubectl get configmap tap-dynamodb-catalog -n v1-sync-helper -o yaml
```

## Regenerate and apply

**Prerequisites:** AWS credentials for the target environment and a working Python/uv environment (see [meltano/README.md](../meltano/README.md)).

With your `kubectl` context pointed at the target cluster, generate and apply the updated catalog:

```bash
aws-vault exec <aws-profile> -- ./scripts/generate-catalog-configmaps.sh \
  -e <meltano-environment> tap-dynamodb | kubectl apply -n v1-sync-helper -f -
```

After applying, the Meltano CronJob will use the updated catalog on its next scheduled run. To pick it up immediately, trigger a manual job run.
